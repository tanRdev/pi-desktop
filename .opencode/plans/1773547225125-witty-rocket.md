# Pi Coding Agent Wiring + Electron Parity Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship an Electron app that uses the real Pi coding agent runtime end-to-end while matching the web shell’s styling and information architecture closely enough that the web app remains a styling sandbox instead of a separate implementation.

**Architecture:** Preserve the existing Electron main/preload/utility-process agent stack and converge the renderer around the shared `PiDeskApi` contract. Make the web shell the visual source of truth, extract duplicated renderer state/feed logic into shared modules, then rebuild the desktop renderer against the web layout/tokens with only thin desktop-specific wrappers for drag regions, live runtime status, and unavailable controls.

**Tech Stack:** Electron 41, React 19, Tailwind v4, Radix UI, TypeScript, pnpm workspaces, Vite/electron-vite, Vitest, Playwright, `@mariozechner/pi-coding-agent`.

---

## Assumptions

- The production runtime stays in `apps/desktop/src/main/*`, `apps/desktop/src/preload/api.ts`, and `packages/agent-host/*`; this project should reuse that path instead of rewriting it.
- The web app in `apps/web` remains a styling/reference sandbox backed by mocks.
- Renderer-driven model switching, agent-mode switching, and persistent multi-thread history do **not** exist in the current shared API/runtime and should not be faked.
- “Looks just like the web app” means matching layout, spacing, typography, tokens, prompt/input behavior, empty states, and inspector IA; desktop-only window chrome/drag-region behavior remains desktop-specific.

## Scope

### In scope
- desktop renderer parity with the web shell
- real Pi agent event flow in the desktop transcript/status surfaces
- shared renderer state/feed extraction to stop web/desktop drift
- shared visual primitives/tokens where they reduce duplication
- capability-gated handling of web-only controls
- validation/test updates for the new renderer structure

### Out of scope
- rewriting `packages/agent-host` or Pi SDK internals
- adding new IPC/runtime commands for model/mode switching
- durable multi-session/thread persistence
- a true file tree/context loader beyond current shell snapshot + activity data
- changing Electron bootstrap/window architecture beyond renderer support changes

## Known constraints / product decisions

- `PiDeskApi` only exposes `shell.getSnapshot`, `agent.getSnapshot`, `agent.prompt`, `agent.reset`, and `agent.subscribe`; the plan should fit that contract unless a later follow-up explicitly expands it.
- The web model selector and agent mode selector are styling-only today. In desktop parity they should render as disabled/read-only or as explanatory status text, not as imaginary behavior.
- The web threads list is styling-only. First-pass desktop parity should show the current active session/thread and map **New Thread** to `agent.reset()`.
- Current Playwright smoke selectors are stale: `tests/e2e/app-launch.spec.ts` expects `data-testid` values that do not exist in the current renderer. The parity work must restore stable selectors or update the test in the same change set.
- Tabs that lack real data (`files`, richer `context`, durable `history`) should show honest empty states backed by real desktop metadata, not fabricated content.

## Risks and mitigations

- **Risk: shared UI/package plumbing becomes heavier than the renderer work itself.**  
  Mitigation: source-alias shared renderer modules in Vite/electron-vite instead of introducing dist-only renderer packaging or unnecessary IPC seams.

- **Risk: desktop visually keeps legacy panels and never fully matches the web shell.**  
  Mitigation: treat the current desktop sidebar/inspector scaffolding as replace/remove work, not as surfaces to “theme toward” parity.

- **Risk: fake controls undermine trust.**  
  Mitigation: render model/mode/thread affordances as disabled/read-only until a real shared API exists; explain why in UI copy/tooltips.

- **Risk: current E2E coverage breaks during the refactor.**  
  Mitigation: add/restore stable `data-testid` hooks while rewriting the renderer and update Playwright in the same chunk.

- **Risk: there is not enough real data for web-style tabs/metrics.**  
  Mitigation: map tabs to `shell.git`, `shell.workspace`, `shell.capabilities`, and `live.activity`; use explicit unavailable/empty states instead of mock metrics.

- **Risk: single-session runtime conflicts with the web shell’s thread list styling.**  
  Mitigation: ship a single active thread representation in this pass and defer durable thread history to a later API/runtime project.

## End-state acceptance checklist

- [ ] Desktop visual layout matches the web shell’s left rail, centered header, empty state, prompt bar, and right inspector structure.
- [ ] Desktop prompt submission uses the real Pi runtime and streams transcript updates into the UI.

- [ ] Shared shell tokens/components are the source of truth for styling, with desktop CSS retaining only platform-specific behavior.
- [ ] Integration tests cover the shared shell-model/feed path and Electron IPC boundary.
- [ ] Playwright smoke passes against the updated desktop UI.
